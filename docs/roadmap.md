# Roadmap：从评估工具到编译生命周期工作台

> 2026-07-04 · 理论框架来自 working paper《编译经济学：智能进步的分层治理框架》（compile-economy-2026-05-20）
> 原则：脑测的 workflow 大多效果不佳，本路线图写成**带 kill criteria 的假设序列**，不是功能承诺清单。每项假设被数据证伪时按预设动作降级或砍掉。

## 定位复述

用编译经济学的语言重述本仓库：企业级 agent 是「把通用能力编译为专用系统」的产物（重 harness + 编译好的脚本），本工作台是编译过程的**度量仪器**。现有两个模块各占生命周期一端：

- 复杂度阶梯 = E3 判准的雏形（该不该编译：穷举成本 vs 训练成本）
- 质量仪表盘 = 编译产物的健康监测（编译得如何）

缺的是中间和之后的环节：漂移检测、重编译触发、验证经济学、真实数据飞轮。以下按优先级排列。

## 缺口一：重编译层（优先级 1，对应 E5）

**假设**：分布漂移是编译产物的主要衰减方式，「重编译频率」可以从已有的 history 数据中提取出可操作的信号。

仪表盘目前是静态快照，而框架的核心动态变量是「分布漂移触发的重编译频率」。这层的接口不依赖真实数据就能先定出来。

最小件：

1. **goldset 时效性指标**——每条 case 增加 `last_verified` 字段，与规则/法规变更日期比对，输出「过期 case 占比」。
2. **指标漂移检测**——基于 `reports/history.jsonl` 的滑动窗口对比（近 N 次 vs 前 N 次），任一指标越级变色即记为漂移事件。
3. **重编译触发报告**——漂移事件 + 过期 case 汇总成一份 `reports/recompile_triggers_<date>.md`，回答「哪些编译产物需要重新验证」。
4. **模式三落地**——「法规时效失效」（法规已废止但 agent 继续引用）从 backlog 升格为这一层的第一个实例：法规库条目带 `effective_until`，扫描 audit_log 中引用已失效法规的 case。

**2026-07-04 范围调整**：设计阶段探查发现，第 4 项依赖的两个前提在 sibling repo（`compliance-review-agent`）均不存在——法规库（`data/regulations/*.md`）是无 ID、无生效日期的纯文本文件；`audit_log` 的 `retrieve` 节点只记录命中条数，不记录命中的具体法规。在前提缺失的情况下做"扫描已失效引用"只会产出假数据，因此本轮第 4 项降级为**契约先行**：只定义 `regulation_refs: [{id, effective_until}]` 字段 schema 和一个 not_instrumented-aware 的扫描器（区分"没装契约"与"装了契约但干净"，避免假绿灯），不做真实扫描。sibling repo 侧的改造（法规库结构化、audit_log 输出引用的法规 ID）列为独立后续项，未来单独排期，不在本轮内。第 1 项同理按数据现状降级：无变更追踪数据源，本轮只做纯 age-based proxy（`last_verified` 距今超 90 天即过期），"与规则/法规变更日期比对"保留为字段方向、本轮不建。完整设计见 [`docs/superpowers/specs/2026-07-04-recompile-trigger-layer-design.md`](superpowers/specs/2026-07-04-recompile-trigger-layer-design.md)。

**Kill criterion**：接入真实运行数据后三个月内，若 history 中测不出任何有意义的漂移信号（全部指标平直），重编译报告降级为被动记录（随 dashboard 附带输出），不做主动报警通道。

## 缺口二：workbench 自身的编译效率（优先级 2，对应 E7）

**假设**：「泛化能力」在本项目里的可操作定义就是接入边际成本；这个成本可测量、且随迭代下降。

评估工具自己也要接受编译效率判准——这是自指但诚实的。

最小件：

1. 每次接入新 agent / 新垂类时，记录**配置行数**与**人时**（`docs/onboarding_log.md`，一次接入一行）。
2. 把「接入成本」作为 workbench 的北极星指标：目标是新 agent 接入 = 只改 `eval/agents.yaml` + goldset 引用，零 Python 改动（现状已接近，需持续验证）。
3. 接入第三个（外部/非自建）agent 时是关键测试——前两个 demo 是和契约共同演化的，第三个才检验契约的泛化性。

**Kill criterion**：若第三个 agent 接入被迫改动 `eval/*.py` 核心逻辑超过一次，说明注册制抽象漏了维度，暂停「泛化」叙事，先重构契约再继续接入。

## 缺口三：验证经济学（优先级 3，对应 E6）

**假设**：模式二 silent failure（置信度校准失真，M013 案例）的大部分实例可以用确定性交叉校验捕获，只有残余需要 LLM 参与。

Workbench 坚持确定性、不调 LLM——这是 compiler 路线的自觉选择，但 M013 证明了它的硬天花板。框架给的出路是「compiler + 低频 runtime verifier」，经济模型随之改变：不再是零 LLM 开销，而是低频 LLM 开销单独记账。

分两步，顺序不可颠倒：

1. **确定性交叉校验先行**——「结论置信度 vs 实际法规依据数量」比对：检索命中 0 条但 confidence > 阈值即标红。不破坏确定性原则，覆盖 M013 类问题。
2. **可选 LLM verifier 车道**——只对步骤 1 抓不住的残余开启；调用次数、token 成本单独记入仪表盘 Cost 维（这本身就是 E6「compiler 退化为 compiler + verifier」的经济学实测）。默认关闭，agent 级 opt-in。

**Kill criterion**：若步骤 1 上线后对新增校准类 case 的捕获率 ≥ 80%，步骤 2 无限期搁置——确定性方案够用时不引入 LLM 依赖。

## 缺口四：真实数据飞轮（优先级 4，依赖真实项目落地）

**假设**：goldset 的说服力最终来自历史失败案例；historical 占比是唯一诚实的进度指标（现状 0%）。

这不是功能，是流程。前三个缺口都是为它预留的管道：

```
生产失败案例 → goldset case（failure_mode + source: historical 标注）
            → 阈值重校准（缺口一的漂移检测消费）
            → 重编译触发（缺口一的报告输出）
            → 下一轮验证（缺口三的校验层把关）
```

最小件：

1. 失败案例录入模板（YAML，与现有 goldset schema 一致，`source: historical` 必填溯源字段）。
2. 仪表盘的 historical 占比作为公开进度条——数字不好看就让它不好看，这个数字本身就是路线图。

**Kill criterion**：不适用——没有真实项目落地时此缺口整体冻结，不做任何投机性建设。

## 排序与依赖

```
缺口一（重编译层）      ─┐ 现在可动，接口先行
缺口二（接入成本自测）  ─┤ 现在可动，随下次接入启动
缺口三（验证经济学）    ─┘ 步骤 1 现在可动；步骤 2 受 kill criterion 门控
缺口四（数据飞轮）      ── 冻结，等真实项目；前三项为其预留管道
```

## 不做什么（继承 README 的边界，用框架语言重述）

- 不做第三个 agent——工作台是编译度量层，不是编译产物。
- 不做仿真沙盒 / GUI 评估 / 默认开启的 LLM-as-judge——与确定性、可审计定位冲突；LLM 只以缺口三步骤 2 的记账式 verifier 形态存在。
- 不在真实数据到来前扩建 goldset 合成 case——合成 case 的边际价值已到平台期，扩建是投机性建设。
