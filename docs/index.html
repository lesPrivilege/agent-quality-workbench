01 — 为什么需要它

立项靠直觉、验收靠 pass/fail，中间是一段没有闭环的空白。

四个具体的缺口，从「拍脑袋」到「有据可依」——每一项都对应工作台里的一个模块。

GAP 01
「该不该上 agent」靠直觉
现状PM 拍脑袋决定，没有结构化评估框架
补什么6 维度加权评分，输出明确的「不建议 / 简单 / 中等 / 复杂」等级
GAP 02
「agent 表现如何」靠跑测试看 pass/fail
现状测试全绿但 M013 这种 silent failure 漏过去了
补什么仪表盘从 audit_log 提取 HITL 触发率、guardrail 拦截率、silent failure 代理等运营指标
GAP 03
两个 agent 项目各自独立评估
现状没有跨项目的横向对比
补什么统一仪表盘，同口径对比两个 agent 的完成率 / 准确率 / HITL 率
GAP 04
立项和验收之间没有闭环
现状立项时说「需要 HITL」，验收时不知道 HITL 率是否合理
补什么复杂度阶梯的推荐等级和仪表盘的实际 HITL 率可以交叉验证

04 — 深度洞察

Silent Failure 的两种正交模式

测试全绿不等于没有问题。真正危险的失败，是那些不报错、静默通过的——它们分成两种正交的模式。

模式一 · 已自动检测

风险信号覆盖缺口

源数据存在关联方/担保/不可逆等结构化风险标记，但 agent 未拦截、静默放行。仪表盘用 contracts.jsonl / materials.jsonl 的结构化字段与 audit_log 的 decision 交叉比对，扫描「有风险信号 + auto_approved」的用例。

0 当前两个 repo：0 命中
模式二 · 规则扫描抓不到

置信度校准失真

数据本身没有结构化风险标记，但模型对依据不足的结论给出了虚高的置信度。问题不在字段里，在模型的自我表达上——规则扫描天然覆盖不到。

M013（量子计算咨询合同）：法规库对该领域返回 0 结果，但初版 prompt 下 LLM 仍给出 confidence = 0.82。原因是模型把「对结论的信心」和「法规依据是否充分」混为一谈。修正 prompt 语义后重验，置信度降至 0.15。

05 — 闭环

复杂度阶梯说「合规审查需要复杂 agent」 → 仪表盘显示 HITL 率 60% → 验证了这个判断。

两个模块互相校验：立项时的推荐等级和运营时的实际指标应该一致。
